## Design Context

### Users
**Homelabbers and DevOps engineers** who manage self-hosted Docker Compose stacks. They open stackd with intent: checking if a deploy succeeded, diagnosing why a stack is unhealthy, triggering a manual sync after a code push. They are technical, busy, and impatient. Primary job: "Is everything up? What broke? Fix it fast." They are not browsing — they are working.

### Brand Personality
**Sharp. Technical. Confident.**

Emotional goal: **In control** — the user should feel like an operator with a clear readout of their infrastructure. Not reassured, not delighted — *powerful and informed*.

Anti-references: Bootstrap boilerplate, consumer SaaS, pastel dashboards, rounded-card Notion-template UIs, anything that looks like a marketing page.

References: **ArgoCD** (stackd is ArgoCD for Docker Compose — information-dense dark dashboard, sidebar navigation, colour-coded health states), **GitHub** (palette discipline, precise typography, restraint with colour). Not the visual style wholesale — the *discipline* behind them.

### Aesthetic Direction
- **Dark only.** Background `#0d1117` (GitHub-dark base). No light mode.
- **Accent colour: electric indigo `#6c63ff`** — distinct from status colours (green/red/yellow/blue are all health states). Indigo reads as "action/brand". Pairs with the logo gradient.
- **Typography: monospace for data, sans-serif for labels.**
  Use `JetBrains Mono` for SHAs, ports, paths, log lines, env keys.
  Use `DM Sans` for all UI labels and body text. `Inter` is banned.
  Keep base size 13–14px. Dense but not cramped.
- **Zero decoration.** No gradients on UI chrome, no drop shadows on cards, no rounded corners above 6px. Borders are 1px solid, never glowing.
- **Status colours are sacred.** Never repurpose green/red/yellow/blue for anything other than health states. Instantly readable at a glance.
- **Motion: minimal and functional.** Only two animations are allowed:
  1. Sync button spin (communicates active work)
  2. Detail panel fade-in (14px translateY, 150ms ease-out — prevents jarring cut)
  All other animations off by default. Always respect `prefers-reduced-motion`.

### Design Principles
1. **Density over decoration** — every pixel must carry information or create structure. White space is used for separation, not aesthetics.
2. **Status first** — health indicators (running/stopped/degraded/error) must be immediately visible without reading text. Colour + shape, not colour alone.
3. **Hierarchy is the UX** — Repo → Stack → Container is the core mental model. The UI must make this hierarchy obvious at every level.
4. **Actions are secondary** — the dashboard is primarily for observation. Action buttons (Sync, Apply, Stop) should be present but visually subordinate to status information.
5. **Technical values deserve technical treatment** — SHAs, ports, paths, timestamps, log lines: always monospace, always muted colour, always exact. Never truncate without a tooltip.
6. **No modal anxiety** — destructive actions need confirmation, everything else shouldn't. Prefer inline state changes over disruptive modal dialogs.
7. **Fail loudly, recover gracefully** — errors are shown immediately with a clear next action. Never silent failures. Never broken empty states.
